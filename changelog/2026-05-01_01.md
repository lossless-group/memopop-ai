# Changelog - 2026-05-01_01

## The Story So Far — memopop-ai From Empty Bun Workspace to Live Desktop Client

This is the first changelog at the **monorepo** level for memopop-ai. We've been writing changelogs inside `apps/memopop-native/changelog/` since the Tauri shell was scaffolded — three of them, all dated 2026-04-27 — but never one at the top of the workspace itself. So this entry catches up the rest, narrates the arc from "Bun workspace with three placeholder app dirs" to "fullscreen desktop client driving the orchestrator's HTTP API live, with milestones, log streaming, file flashing, stop button, and a self-healing sidecar manager," and gives the next changelog a place to start.

If you want the granular foundation, read those three first:

- [`apps/memopop-native/changelog/2026-04-27_03.md`](../apps/memopop-native/changelog/2026-04-27_03.md) — Full onboarding journey: outline gallery, firm creation, deal capture.

## The Arc, in Order

1. **2026-04-27.** Three changelogs in one day. Tauri shell stood up, plugins installed, transport interface defined, `LocalTransport` calling a single Rust `api_dispatch` command, GET /firms wired end-to-end. Then Settings + firm picker. Then full onboarding flow — gallery, outline detail modal, firm creation modal with snake_case slug preview and idempotent `.gitignore` patching, deal capture form. Stopped at "Ready to generate" placeholder.

2. **2026-04-30.** Two things happened in parallel:
   - `JourneyBreadcrumbs` shipped (above the gallery / between header and content) — a four-step progress bar (Outline → Firm → Deal → Generate) with backward navigation.
   - The orchestrator gained a FastAPI sidecar at `investment-memo-orchestrator/src/server/`. The desktop client now had something to talk to. The sidecar's API mirrors `generate_memo()` kwargs.

3. **2026-04-30 → 2026-05-01 (overnight).** Wired the desktop to the sidecar.
   - **Rust:** `Cargo.toml` gained `reqwest` + `tokio[time]`. New `src/api/sidecar.rs` with `SidecarManager` (lazy spawn from `{repoPath}/.venv/bin/python`, healthz polling, child handle in `Mutex<Option<CommandChild>>`, `forward()` for JSON, `shutdown()` for SIGKILL). New match arms in `router.rs` for `/memos*` (POST + status + list + artifacts) — all routed through `forward_to_sidecar` which strips the dispatcher-only `repoPath` field before forwarding. `lib.rs` registers the manager in app state and wires shutdown to `WindowEvent::Destroyed`.
   - **Frontend:** `Transport` interface gains `subscribeEvents(jobId, onEvent)`. `LocalTransport.subscribeEvents` opens an `EventSource` direct to `http://127.0.0.1:8765/memos/{id}/events` — bypassing the Rust dispatcher entirely. New `running_job` flow stage with milestones + events arrays. `DealCreationModal` now does a real `POST /memos`. New `JobLogViewer` modal with three tabs: Milestones, Files, Logs.

4. **2026-05-01 morning.** Built `MilestoneTimeline` and `ArtifactBrowser` (live polling of artifacts every 3s, files flash green on arrival). The orchestrator's `MilestoneExtractor` published structured `milestone` events; the timeline rendered them as a vertical chronological list.

5. **2026-05-01 (the user-feedback inflection).** First end-to-end run with an actual memo. Four issues surfaced:
   1. Log viewer ate memory and the screen went black on a 30s cadence.
   2. The checklist updated only at the end (consequence of #1).
   3. Files landed at `output/ChromaDB-v0.0.1/` — wrong dir.
   4. A second run silently overwrote the first run's directory.

6. **2026-05-01 evening — the redesign.** Replaced the tabbed `JobLogViewer` modal with `JobView`: fullscreen, three columns simultaneously visible (`PhaseChecklist` · `LogStream` · `ArtifactBrowser`), no tabs. The user's words: *"this will be THE THING THEY ARE TRYING TO DO, WITH IMPORTANCE."* Fullscreen made it that. Deleted the old `JobLogViewer.svelte` and `MilestoneTimeline.svelte` — the new components own that space cleanly.

7. **2026-05-01 late evening — the hardening pass.** Performance fixes (events array capped at 2,000 in store, rendered lines capped at 1,000 with a "earlier lines hidden — full log on disk" banner, `requestAnimationFrame`-throttled auto-scroll). Stop button (red `⏹ Stop` in the JobView header, calls `POST /actions/stop-sidecar` which routes to `SidecarManager::shutdown()` directly so it works when Python is wedged). Tauri shell extended to handle `WindowEvent::CloseRequested` alongside `Destroyed` (Cmd+W and Cmd+Q both reliably terminate the sidecar). `SidecarManager::ensure_running` self-heals — always probes `/healthz` first, drops stale handles, spawns fresh.

8. **Where we are now (commit time).** The user can launch MemoPop, anchor an orchestrator, click an outline, set up a firm, fill a deal form, click Generate, and watch:
   - The phase checklist tick over in real time.
   - Log lines stream in the dark center pane with a blinking purple cursor.
   - Files flash green on the right as they appear in the firm-scoped output dir.
   - A red Stop button kills the run cleanly mid-flight.
   - Quitting the window kills the sidecar.

### How a request flows

```
┌─────────────────────────────────────────────────────────────────┐
│  Webview (Svelte runes UI)                                      │
│                                                                 │
│  getTransport().request('POST', '/memos', { ... })              │
│  getTransport().subscribeEvents(jobId, onEvent)                 │
└────┬───────────────────────────────────────────┬────────────────┘
     │ JSON via Tauri invoke                     │ EventSource (HTTP/SSE)
     ▼                                           │
┌─────────────────────────────────────────┐      │
│  Rust dispatcher (api_dispatch)         │      │
│  - Routes /firms, /outlines, /actions   │      │
│  - Forwards /memos* via reqwest →       │      │
│    SidecarManager::forward()            │      │
└────┬────────────────────────────────────┘      │
     │ HTTP localhost:8765                       │
     ▼                                           │
┌─────────────────────────────────────────┐      │
│  FastAPI sidecar (Python, in venv)      │◄─────┘
│  - generate_memo() in a worker thread   │
│  - JobEventBus → SSE                    │
│  - persist_job_logs at terminal         │
└─────────────────────────────────────────┘
```

The split is deliberate: the Rust dispatcher is the right place for JSON request/response (uniform error shape, single Tauri command surface, easy to swap for HTTP-mode later). SSE is browser-native — the webview opens an `EventSource` directly. Reimplementing chunked HTTP forwarding inside `api_dispatch` would have been wasted code.

## Design Decisions Worth Recording

**The Transport interface has exactly two methods.** `request()` for JSON request/response, `subscribeEvents()` for SSE streaming. Resisted the temptation to add a third (`watchFiles`, `subscribeStatus`, etc.) — anything periodic can be polled via `request()` and a `setInterval`, anything streaming can be expressed as SSE. Two methods stay easy to swap when local mode becomes hosted mode.

**The Tauri sidecar lives in the orchestrator's repo, not memopop-ai's.** Documented in `context-v/explorations/Moving-an-Agent-Orchestrator-to-an-API.md`. The FastAPI is a thin import wrapper around `generate_memo()`; it has to live next to the Python source it imports. The orchestrator gains an HTTP surface for any non-Python client; we gain a memopop-specific UI; nobody owns two copies of the workflow.

**SSE goes direct from webview to localhost:8765.** Considered routing through Rust for "uniformity," rejected because (a) `EventSource` is browser-native and works perfectly, (b) the FastAPI CORS allowlist already covers Tauri origins (`tauri://localhost`, `http://tauri.localhost(:port)?`), (c) Rust would have to forward chunked HTTP responses, which is real engineering for no real benefit.

**Lazy sidecar spawn.** `SidecarManager` doesn't start anything until the first `/memos*` call. Spawn cost (250ms-1s) is paid once, on the first action that needs it. The user clicking around the gallery and reading outline detail modals never wakes up Python.

**Self-healing instead of version-checking.** `ensure_running` always probes `/healthz` first. If the sidecar is dead, hung, or running stale code (which we can't directly detect without instrumentation), the simplest reliable approach is "if healthz is silent, kill any handle we have and spawn fresh." Future work could add a `/healthz`-returned version hash for staleness detection; today's bias is "self-heal anything that isn't responding."

**Stop bypasses Python.** `POST /actions/stop-sidecar` routes to `SidecarManager::shutdown()` in Rust directly — does not round-trip through the FastAPI server. The whole point of "Stop" is that Python may be wedged. SIGKILL via stored `CommandChild` is the only design that works when the sidecar is in a deadlocked retry loop or rate-limited. SIGKILL means cancelled runs lose their structured log file (no time for `persist_job_logs` to run); for v1 that trade-off is acceptable.

**The job UI is a route-level page, not a modal.** Earlier iteration was a centered modal capped at 760px. The first user-test feedback was decisive: *"most people will likely not have it as a small part of their screen."* Replaced with a full-viewport three-column layout. Logs dominate, milestones are glanceable on the left, files flash on the right.

**Performance bounds at the store level, not just the view level.** Capping rendered lines fixes layout thrash; but the events array itself was the source of GC stalls (each event a deep-reactivity proxy). Cap at the store with `slice(-2000)` whenever the array exceeds the bound. View-level caps are a secondary defense.

**Flow state is a discriminated union, not a stack.** `idle | outline_detail | create_firm | create_deal | ready_to_run | running_job`. One variable; conditional rendering in `+page.svelte` reads cleanly. Resisted earlier urge to model a modal stack (since the spec's flow is linear, not nested).

## What's Next

The journey from outline → memo is closed end-to-end for the local-sidecar mode. Three categories of follow-up, none urgent:

1. **Polish the export layer.** The orchestrator produces a 67k-character markdown memo plus a one-pager HTML/PDF plus a Word doc. The desktop client currently shows the output directory path and the file list — but doesn't open files. A "View memo" button that opens `7-{Company}-vX.Y.Z.md` in the system viewer (or a built-in markdown preview) would close the loop visibly. `tauri-plugin-shell::open()` is the move.

2. **Branded export flow.** The orchestrator's `export-branded.py` produces firm-styled HTML/PDF using the `brand-{firm}-config.yaml` we scaffold-stub during firm creation. Today the brand config has only `company.conventional_name` populated. A "Brand setup" wizard (hinted at in `context-v/specs/An-Onboarding-User-Journey-for-Memopop-Native.md`) is the natural next feature — promising version is the agent-driven version that auto-fetches colors/fonts/logos from a firm's website URL.

3. **Hosted mode.** Today's transport is `LocalTransport`. Tomorrow's is `HttpTransport` to a deployed FastAPI. Both implement the same two-method interface; the components don't change. The work between local and hosted is auth, multi-tenant isolation, persistent storage, and a job queue — none of which are needed before the first hosted user.

The `memopop-site/` and `memopop-web-app/` apps in this workspace are placeholders for the hosted future. They'll get their own changelog entries when there's something to say.
