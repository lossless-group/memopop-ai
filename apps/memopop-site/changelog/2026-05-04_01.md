## What's live

`https://lossless-group.github.io/memopop-ai/` — the canonical URL going forward. Default org-pattern URL (`<org>.github.io/<repo>/`), no custom domain. The Lossless Group's project-page habit going forward is exactly this pattern: every project gets a GitHub-hosted page, every project's URL is predictable from the repo name.

The previous `index.astro` was a generic marketing-template Astro page (white background, navy/teal corporate palette, placeholder pricing tiers, fake "50+ VC firms"). It's been replaced wholesale.

The replacement is a structural port of `site/src/pages/projects/gallery/memopop-ai.astro` from the consulting site, redrafted to stand alone:

The full 9-step pipeline with cast-of-characters preamble (Extractor → Researcher → Writer → Validator → Editor → Versioned Output) and the Orchestrator wrapping all of them.

## The /changelog hub — the actual interesting bit

Four content collections, one schema, one timeline.

```ts
// src/content.config.ts
const changelogSchema = z.object({
  title: lenientString,
  lede: lenientString,
  summary: lenientString,
  date: lenientDate,
  date_authored_initial_draft: lenientDate,
  // ... ten more optional date fields ...
  category: lenientString,
  status: lenientString,
  at_semantic_version: lenientString,
  augmented_with: lenientString,
  tags: lenientStringArray,
  authors: lenientStringArray,
  // ...
}).passthrough();

const monorepo     = defineCollection({ loader: glob({ pattern: '*.md', base: '../../changelog' }),                schema: changelogSchema });
const orchestrator = defineCollection({ loader: glob({ pattern: '*.md', base: '../memopop-orchestrator/changelog' }), schema: changelogSchema });
const site         = defineCollection({ loader: glob({ pattern: '*.md', base: './changelog' }),                    schema: changelogSchema });
const native       = defineCollection({ loader: glob({ pattern: '*.md', base: '../memopop-native/changelog' }),    schema: changelogSchema });
```

### Why "permissive" was the load-bearing word

A survey before writing the schema turned up four *very* different frontmatter shapes:

| Source | Date field | Tags style | Frontmatter at all? |
|---|---|---|---|
| Orchestrator (newer) | `date_authored_initial_draft` | YAML flow `[Foo, Bar]` | Yes, rich |
| Orchestrator (older) | (none — the file just starts with `# Changelog`) | n/a | **No** |
| Native | `date` (single field) | YAML list | Yes |
| Site | `date_authored_initial_draft` + `usage_index` + `publish` + `app` + `files_added` | YAML flow | Yes |
| Monorepo | `date_authored_initial_draft` | YAML flow | Yes |

The schema accepts all of it — every field is `lenientString` / `lenientDate` / `lenientStringArray` (a `z.preprocess` wrapper that coerces empty strings and `null` to `undefined` *before* type-checking), and the schema is `.passthrough()` so unknown fields don't trigger validation errors. The 8+ early orchestrator entries that have *zero* frontmatter render with title falling back to the slug (`2025-11-16_01`) and date parsed from the slug's `YYYY-MM-DD` prefix.

### Routes

- `/changelog` — unified timeline, newest-first, source-filter pills with counts (Orchestrator 60 / Native 5 / Site 2 / Monorepo 1).
- `/changelog/[source]/[...slug]` — one per entry. `[...slug]` (rest segment) is used so future nested paths in any source's changelog folder won't break the route.

### Sorting

```ts
function pickDate(data, id) {
  for (const key of [
    'date_authored_initial_draft', 'date', 'date_first_published',
    'date_authored_current_draft', 'date_created', 'date_modified',
    'date_last_updated',
  ]) {
    const v = data[key];
    if (v instanceof Date && !Number.isNaN(v.getTime())) return v;
  }
  return deriveDateFromId(id);  // YYYY-MM-DD prefix from filename
}
```

`Date` formatting on the rendered pages explicitly sets `timeZone: 'UTC'` — a `2025-11-16` ISO date parses as UTC midnight, then `toLocaleDateString()` shifts to local TZ and silently displays the day before. Caught on the first render-and-refresh pass.

## The orchestrator submodule

`apps/memopop-orchestrator/` is a git submodule pointing at `lossless-group/investment-memo-orchestrator`. The orchestrator is its own repo, lived on its own release cadence, and was sitting at `ai-labs/investment-memo-orchestrator/` as a peer to `memopop-ai/`. Now it *also* lives inside the monorepo, where:

- `apps/memopop-site/src/content.config.ts` reads the orchestrator's `changelog/*.md` directly
- `git submodule update --remote apps/memopop-orchestrator` pulls upstream changes
- The deploy workflow's `actions/checkout@v4` uses `submodules: true` so CI sees the same content

Submodule was chosen over `git subtree add` (which would fold history into this repo and complicate pushing back upstream) and over a plain `cp -R` (which would lose the upstream link entirely). The trade-off is that contributors cloning fresh need `git clone --recurse-submodules` or a follow-up `git submodule update --init` — documented for now via the workflow file as a reference.

The orchestrator's `changelog/releases/` subfolder (release-notes files like `v0.4.1-release-notes.md`) is intentionally excluded — the glob pattern is `*.md`, not `**/*.md`. Different shape, can be added as its own collection later if useful.

## The GuiCallout component

Cross-page CTA aside that asks readers "Don't want to run LangChain through the command line?" and routes them to the consulting-site project page. Lives on the home page (after the CLI section) and on every changelog page (between header and timeline on the index, after the body on detail pages).

Three iterations on the layout in this session:

1. **First pass:** single-column, text-only.
2. **Second pass:** two-column 1fr/320px (text left, screenshot right).
3. **Final:** two-column 1fr/2fr — the bordered "box" wraps only the text + CTA, the screenshot floats free at 2/3 of the container with its own subtle border + soft shadow + cyan-tinted hover lift.

The screenshot is now click-to-expand using a native `<dialog>` element:

```astro
<dialog class="callout-lightbox" id={dialogId}>
  <button data-lightbox-close>✕</button>
  <img src={screenshotUrl} class="lightbox-image" loading="lazy" />
</dialog>
```

`<dialog>` chosen over a custom modal because it ships with Escape-to-close, focus trap, top-layer rendering, and `::backdrop` pseudo-element styling for free. The wiring script binds via `data-lightbox-trigger` / `data-lightbox-target` (a per-instance random ID), so multiple GuiCallouts on the same page won't collide. Body scroll locks while the lightbox is open. Clicking the backdrop, the ✕ button, or the image itself all close it.

A `loading="lazy"` on the lightbox image means the full-size webp isn't fetched until the dialog opens — important since the callout appears on 67+ detail pages.

## The deploy pipeline

`.github/workflows/deploy-site.yml`:

```yaml
on:
  push:
    branches: [main]
    paths:
      - 'apps/memopop-site/**'
      - 'packages/shared-styles/**'
      - '.github/workflows/deploy-site.yml'
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: true            # pulls memopop-orchestrator
      - uses: oven-sh/setup-bun@v2
      - run: bun install --frozen-lockfile
      - run: bun run --filter memopop-site build
      - uses: actions/configure-pages@v5
        with:
          enablement: true            # see "gotcha" below
      - uses: actions/upload-pages-artifact@v3
        with:
          path: apps/memopop-site/dist
  deploy:
    needs: build
    environment: github-pages
    steps:
      - uses: actions/deploy-pages@v4
```

Path filters mean the workflow only fires when the site or its declared dependencies change — pushes that touch the orchestrator (via submodule bump) or the desktop client don't trigger redeploys until the site itself is updated.

### The two gotchas worth recording

**1. The `base: '/memopop-ai/'` in `astro.config.mjs`.** Default org-pattern URL is `https://<org>.github.io/<repo>/`. Astro needs to know about that subpath at build time so generated asset URLs (`<img src="/_astro/...">`) resolve. The `BaseLayout.astro` favicon link uses `import.meta.env.BASE_URL` so it picks up the prefix automatically; every internal link in `index.astro` and the changelog pages does the same. If a custom domain ever gets added, both `site` and `base` change in one place.

**2. The `.nojekyll` file in `public/`.** GitHub Pages will Jekyll-process any directory by default, which silently drops files starting with `_`. Astro's bundled assets land in `_astro/`. An empty `.nojekyll` opts out and is mandatory for any non-Jekyll Pages site.

### And the gotcha that shipped this commit four times

`actions/configure-pages@v5` failed three times with `Resource not accessible by integration` before the deploy succeeded. The error meant Pages wasn't enabled on the repo yet, and `enablement: true` (which I added in commit `098feb5`) couldn't enable it itself because the org's `GITHUB_TOKEN` permission policy doesn't grant Pages-enable to integrations. The fix was the manual UI toggle: **Settings → Pages → Source = "GitHub Actions"** — a one-time setup per repo. After flipping that, a `gh run rerun` on the same commit succeeded. `enablement: true` was left in the workflow on the theory that future repos with more permissive token policies will benefit and current behavior is unchanged.

## What's next, in rough order

1. **Fonts.** The `Header` and `Footer` components from the consulting site weren't ported. Fine for now, but the splash page will eventually want a top nav. Decide whether to keep the page footer-only or add a minimal top nav with `MemoPop AI` (logo) + `Changelog` + `GitHub`.
2. **Releases collection.** The orchestrator's `releases/*.md` files are excluded from the changelog hub. They have a different shape (no dated filename, content is release-note narrative). Worth surfacing eventually as a separate `/releases` route.
3. **Image rendering for changelog entries.** Many entries have `image_prompt` and `image` frontmatter fields. Right now neither is used by the renderer. A header banner per entry would lift the visual weight of long technical entries.
4. **Pattern documentation.** The directions baked into this work — `apps/<name>-site/`, the `.nojekyll` + base-path setup, the deploy workflow shape — should become a short README at the monorepo root for the next project that adds a Pages site.
5. **Deprecation cleanup.** The Actions deprecation warning (`Node.js 20 actions are deprecated`) applies to `actions/checkout@v4`, `actions/configure-pages@v5`, `actions/upload-artifact@v4`. June 2nd, 2026 is the forced-default date. Bump versions when releases are out, or set `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24=true` to opt in early.
